What Are Vectors?

From Data to Numbers

A vector is a fixed-size array of floating-point numbers that captures the semantic meaning of data. 

"wireless headphones"  →  [0.12, -0.84, 0.37, ... ]
"bluetooth earbuds"    →  [0.11, -0.81, 0.39, ... ]
"kitchen blender"      →  [-0.72, 0.45, -0.13, ... ]

Similar items have nearby vectors. Unrelated items are far apart.

How Are Vectors Created?

  • Embedding models (LLMs, Ollama, OpenAI) convert text to vectors
  • Image models (CLIP, ResNet) convert images to vectors
  • Custom features — hand-crafted numeric attributes

A Semantic View on Data

Traditional databases search by exact match — a keyword either matches or it doesn't.

Vector search finds items by meaning. A query for "affordable laptop" finds "budget notebook computer" even though no words overlap.

Exact match: "laptop" = "laptop"
Semantic match: "laptop" ≈ "notebook" ≈ "ultrabook"

Use Cases

  • Semantic search — find by meaning, not keywords
  • Recommendations — "customers also liked"
  • RAG — retrieval-augmented generation for LLMs
  • Anomaly detection — find outliers in data
  • Image / face similarity — visual search
  • Deduplication — find near-duplicates at scale

What is the GigaMap JVector Integration?

GigaMap

  • Large-scale indexed collection for billions of entities
  • Lazy-loaded segments with multiple index types
  • Persisted via EclipseStore — no ORM, no serialization overhead
  • Supports Bitmap, Lucene, and JVector indices

JVector

  • High-performance HNSW vector similarity search library
  • By DataStax — pure Java, no native dependencies
  • SIMD-accelerated via Panama Vector API (Java 20+)
  • Supports in-memory and on-disk graph indices

Together: persistent vector search that survives restarts, scales to millions of vectors, and queries in sub-millisecond time.

Core Concepts

Vectorizer<E>

Converts entities to float[] vectors. Two modes:

Embedded

Vector already in entity — no duplicate storage

Computed

Vector from external API — stored separately for persistence

Similarity Functions

  • COSINE — direction-based, best for text/semantic search
  • DOT_PRODUCT — fast for pre-normalized vectors
  • EUCLIDEAN — absolute distance, geographic/spatial data

HNSW Parameters

  • dimension — vector size (e.g. 384, 768)
  • maxDegree — graph connections per node
  • beamWidth — search width during build

Higher values = better recall, more memory

How HNSW Search Works

Hierarchical Layers

Layer 2 — sparse (entry point)
▼ navigate to nearest
Layer 1 — medium density
▼ refine candidates
Layer 0 — all nodes (fine-grained)
▼
Top-k results

Search Algorithm

  • Enter at top layer (few nodes)
  • Greedily navigate to closest node
  • Descend layers, expanding candidate set
  • At bottom layer, return top-k by score

Why it's fast

  • Sub-linear query complexity: O(log n)
  • Skip-list-like layer navigation
  • SIMD-accelerated distance computations

Basic Setup

// 1. Create a GigaMap and register the VectorIndices category
GigaMap<Product> gigaMap = GigaMap.New();
VectorIndices<Product> vectorIndices =
    gigaMap.index().register(VectorIndices.Category());

// 2. Configure the HNSW index
VectorIndexConfiguration config = VectorIndexConfiguration.builder()
    .dimension(384)                                      // embedding size
    .similarityFunction(VectorSimilarityFunction.COSINE)   // best for text
    .maxDegree(16)                                       // graph connectivity
    .beamWidth(100)                                      // build quality
    .build();

// 3. Create the index with a Vectorizer
VectorIndex<Product> index = vectorIndices.add(
    "product-embeddings",   // index name
    config,
    new ProductVectorizer()  // entity → float[]
);

The Vectorizer

Embedded Mode

// Vector already stored in the entity
class ProductVectorizer
    extends Vectorizer<Product>
{
    public float[] vectorize(Product p)
    {
        return p.embedding();
    }

    public boolean isEmbedded()
    {
        return true;
    }
}

No duplicate storage. Reads vectors directly from entities.

Computed Mode default

// Vector computed via external API
class ProductVectorizer
    extends Vectorizer<Product>
{
    private final EmbeddingModel model;

    public float[] vectorize(Product p)
    {
        return model.embed(
            p.description()
        );
    }
    // isEmbedded() defaults to false
}

Vectors persisted separately. No recomputation on restart.

Advanced Features

On-Disk Storage

VectorIndexConfiguration.builder()
  .dimension(768)
  .onDisk(true)
  .indexDirectory(Path.of("/data/vectors"))
  .enablePqCompression(true)
  .pqSubspaces(48)
  .build();
  • Memory-mapped graph for datasets > RAM
  • PQ Compression reduces memory further
  • Reranking with exact vectors preserves accuracy

Background Processing

VectorIndexConfiguration.builder()
  .dimension(768)
  .eventualIndexing(true)
  .optimizationIntervalMs(60_000)
  .persistenceIntervalMs(30_000)
  .persistOnShutdown(true)
  .build();
  • Eventual Indexing — async graph mutations
  • Background Optimization — periodic cleanup
  • Auto-Persistence — save at intervals

Choosing a Similarity Function

Function Formula Best For Range
COSINE cos(θ) = (A · B) / (||A|| × ||B||) Text embeddings, semantic search, CLIP [-1, 1]
DOT_PRODUCT A · B = Σ(Ai × Bi) Pre-normalized vectors, recommendations (-∞, +∞)
EUCLIDEAN ||A - B|| = √Σ(Ai - Bi)² Geographic data, image pixels, FaceNet [0, +∞)

When in doubt, use COSINE — the most robust default for embedding-based search.
