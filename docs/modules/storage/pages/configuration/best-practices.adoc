= Best Practices: Dev / Test / Staging / Prod

{product-name} ships with conservative, frictionless defaults so that getting started is a one-liner.
A number of features are intentionally *off by default* — turning them on globally would be a breaking change for existing applications, and most of them carry a small operational cost (extra disk space, extra setup, slower startup) that is only worth paying once you leave a developer's machine.

This page lists those features and gives a recommendation per environment.
Each section links to the canonical configuration page that documents *how* to enable the feature; this page only documents *whether* and *why* you should.

NOTE: Recommendations are guidelines, not rules.
The right setting depends on your workload, your tolerance for downtime, and your operational constraints.
When in doubt, prefer the safer setting in production and the more convenient one in development.

[IMPORTANT]
====
Keep the *goal* of each test type in mind when picking settings.
A unit test exercising business logic gets nothing from a real lock file, continuous backup, or production channel count — those are noise that slow the test down and obscure failures.
A staging run validating a release, by contrast, exists *precisely* to exercise those operational features end-to-end; turning them off there defeats the point of staging.

The `Test` column covers tests whose target is application logic; the `Staging` column covers tests whose target is the production deployment itself.
If a single test suite mixes both goals, split it — or pick the column whose target dominates and accept that the other goal is being checked elsewhere.
====

== Per-Feature Recommendations

[#lock-file]
=== Lock file

*Default:* off. *See:* xref:configuration/lock-file.adoc[Lock File].

{product-name}'s proprietary lock file prevents two {product-name} processes from operating on the same storage concurrently, which would corrupt the data files.
The OS-level file lock is sufficient for the common case of a single application owning the storage; the proprietary lock file is only needed when more than one {product-name} application may try to access the same files.

[cols="1,3"]
|===
| Dev | Off. Quick restarts from an IDE produce stale-lock waits with no benefit when only one process accesses the storage.
| Test | Off. Same as dev. Enable only in tests that deliberately exercise multi-process access.
| Staging | Match prod. If prod runs with the lock file on, staging must too — otherwise staging is not exercising the same startup and restart paths as prod.
| Prod | On if more than one JVM may share the storage (for example blue/green deployments, an accidental double-start during a rolling restart, or a sidecar process). Off is safe if a single application owns the storage exclusively.
|===

[#deletion-directory]
=== Deletion directory

*Default:* off. *See:* xref:configuration/storage-files-and-directories.adoc[Storage Files and Directories].

When configured, the housekeeping process moves data files into the deletion directory instead of erasing them.
Recovering an object that was incorrectly garbage-collected (for example because a root reference was unintentionally dropped before a `store()`) is impossible after a normal delete, but trivial as long as the file still exists in the deletion directory.

[cols="1,3"]
|===
| Dev | Off. Disk usage and operational complexity outweigh the benefit on a developer machine where the storage can be recreated.
| Test | Off. Recreate fixtures rather than recover them.
| Staging | *On.* Mirror prod so the cleanup job, disk allocation, and any recovery procedures are exercised before they matter.
| Prod | *On.* Cheap insurance against operator error and against bugs in storing logic. Pair with a periodic cleanup job (or a generous disk allocation).
|===

[#truncation-directory]
=== Truncation directory

*Default:* off. *See:* xref:configuration/storage-files-and-directories.adoc[Storage Files and Directories].

Files that get truncated during housekeeping (for example when a file's used-ratio drops below the configured threshold and it is rewritten in compacted form) are copied to this directory before truncation.
Useful for forensics if you suspect housekeeping is removing live data, but rarely needed in healthy operation.

[cols="1,3"]
|===
| Dev | Off.
| Test | Off, unless the test under investigation specifically exercises file truncation.
| Staging | Optional. Same default as prod — leave off unless reproducing a housekeeping-related incident.
| Prod | Optional. Enable temporarily during incident investigation; leave off by default to control disk usage.
|===

[#continuous-backup]
=== Continuous backup

*Default:* off. *See:* xref:configuration/backup/continuous-backup.adoc[Continuous Backup].

When enabled, every change written to the primary storage is mirrored to a backup directory in real time.
The backup is byte-identical to the primary and can be used directly as a storage if the primary is lost.
Overhead is modest because writes go to two targets in parallel rather than being copied after the fact.

[cols="1,3"]
|===
| Dev | Off. The primary storage is disposable.
| Test | Off, unless the test specifically exercises backup/restore behaviour.
| Staging | *On.* The backup pipeline itself is part of what staging validates — running staging without it leaves a hole in the dress rehearsal.
| Prod | *On.* The single most effective protection against disk failure or accidental deletion of the primary storage. Place the backup on a separate physical disk, host, or xref:storage-targets/index.adoc[storage target].
|===

[#full-backup]
=== Full backup

*Default:* not applicable (manual operation). *See:* xref:configuration/backup/full-backup.adoc[Full Backup].

A full backup is a one-shot snapshot triggered via `EmbeddedStorageManager.issueFullBackup(...)`.
It is a *blocking* operation: store operations are paused for the duration, which can be significant for large storages.

[cols="1,3"]
|===
| Dev | Ad-hoc. Use to capture interesting fixtures or before destructive experiments.
| Test | Ad-hoc. Useful for seeding integration-test environments from a known state.
| Staging | Scheduled, on the same cadence as prod. The blocking pause is itself a load-test signal — if it disrupts staging it will disrupt prod.
| Prod | Schedule during off-peak hours (for example a nightly cron) as a periodic safety net on top of <<continuous-backup,continuous backup>>. Also recommended before major migrations or schema changes.
|===

[#adaptive-housekeeping]
=== Adaptive housekeeping

*Default:* off (fixed time budget of 10 ms per 1-second interval). *See:* xref:configuration/housekeeping.adoc[Housekeeping].

Adaptive housekeeping increases the time budget when the housekeeping process cannot keep up with incoming writes.
Without it, write-heavy workloads can leave files uncollected and cause disk usage to grow well past what the data warrants.

[cols="1,3"]
|===
| Dev | Off. The fixed default keeps housekeeping unobtrusive on a developer machine.
| Test | Off. Predictable, repeatable behaviour is more valuable in tests than throughput.
| Staging | Match prod. If prod runs adaptive housekeeping, staging must too — otherwise staging's disk usage and pause behaviour diverge from production under load.
| Prod | On for write-heavy workloads. The fixed default is sufficient for read-heavy or low-write applications; switch to adaptive when you observe disk usage growing faster than the underlying data.
|===

[#channel-count]
=== Channel count

*Default:* 1. *See:* xref:configuration/using-channels.adoc[Using Channels].

Channels are the IO threads (and corresponding storage subdirectories) that {product-name} uses in parallel.
The channel count must be a power of two and is fixed for the lifetime of a storage — changing it later requires migration.
Pick the right number up front.

[cols="1,3"]
|===
| Dev | 1. Simplest setup, easiest to inspect on disk.
| Test | 1, unless the test specifically exercises multi-channel parallelism.
| Staging | Use the same channel count you intend for prod. Channel count cannot be changed without migration, so staging is where you confirm the choice on representative hardware and workload.
| Prod | Tune to your workload and hardware. More channels mean higher write parallelism but more files and more memory; a single channel is often fine for read-heavy or moderate-write applications.
|===

[#read-only-mode]
=== Read-only mode

*Default:* off. *See:* xref:configuration/readonly.adoc[Read-only Manager].

A read-only storage manager rejects all writes and disables housekeeping, which lets it coexist with another writing manager — the only supported way to have two managers on the same storage.
Useful for read replicas, ad-hoc inspection of a backup, or analytics workloads that should not mutate data.

[cols="1,3"]
|===
| Dev | Not applicable.
| Test | Optional. Useful for tests that load a fixture and assert against it without ever mutating.
| Staging | On for replica or snapshot-inspection deployments — the same scenarios as prod, exercised against staging data so the read-only manager's lifecycle is validated before it runs against production data.
| Prod | On for read-replica or snapshot-inspection deployments. Keep the read-only manager short-lived: it does not see writes from the primary manager and will eventually be invalidated by housekeeping there.
|===

[#jmx-monitoring]
=== JMX monitoring

*Default:* JMX beans are always exposed; remote JMX is configured at the JVM level and is off by default. *See:* xref:misc:monitoring/index.adoc[Monitoring].

{product-name} publishes JMX beans for storage growth, entity cache, lazy references, object registry, and per-channel metrics.
The beans themselves are free to expose; the operational risk is in how the JVM exposes the JMX port.

[cols="1,3"]
|===
| Dev | Auth and SSL off (typical local-only JMX setup) is fine.
| Test | Auth and SSL off is fine in isolated test infrastructure.
| Staging | *Auth and SSL on*, configured exactly like prod. Staging is reachable from more than just a developer's laptop, and the JMX hardening recipe needs to be exercised end-to-end before it's relied on in prod.
| Prod | *Authentication and SSL must be on.* Restrict the JMX port to a monitoring network or block it at the firewall. An unauthenticated JMX port is remote code execution.
|===

[#rest-interface]
=== REST interface

*Default:* off (`org.eclipse.store.rest.enabled=false` in the Spring Boot integration). *See:* xref:rest-interface/setup.adoc[REST Interface Setup].

The REST interface exposes the object graph for inspection and is invaluable during development and debugging.
It is read-only at the protocol level, but the data it returns is your application's data, so the same access controls that govern the application as a whole must govern the REST endpoint.

[cols="1,3"]
|===
| Dev | On. Useful for inspecting the live object graph from a browser or the client GUI.
| Test | Optional. Convenient for diagnosing test failures.
| Staging | Off by default, or behind the same auth and network isolation as prod. The point of staging is to confirm those controls work — leaving the REST interface wide open in staging defeats it.
| Prod | Off by default. If you need it for operations, place it behind authentication (Spring Security or equivalent) and restrict it to an internal network. The bundled Client GUI is a development tool and should not be exposed publicly.
|===
